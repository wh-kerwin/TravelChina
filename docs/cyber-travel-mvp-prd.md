# 赛博旅游 App MVP PRD

## 1. 文档目标

本文档用于定义“赛博旅游 App”首版 MVP 的产品目标、核心功能、页面结构、内容范围和验收标准，帮助后续的产品设计、视觉设计、3D 资产制作和技术开发在同一范围内推进。

## 2. 产品概述

### 2.1 产品定位

赛博旅游 App 是一个以“卡通化中国城市与地标景点漫游”为核心体验的内容型应用。用户通过全国地图进入城市，再从城市页浏览代表性景点，并进入单个景点的详情页进行视觉化游览。

首版的重点不是完整覆盖全国，也不是高精度还原所有景点，而是验证以下体验是否成立：

- 用户是否愿意通过地图探索城市
- 用户是否会点击并浏览景点详情
- 卡通视觉 + 轻量 3D 展示是否足够形成差异化体验

### 2.2 目标用户

- 对国内旅游、地标建筑、城市文化感兴趣的普通用户
- 对卡通风格、轻量互动体验接受度较高的年轻用户
- 有亲子教育、地理启蒙、历史文化轻学习需求的用户

### 2.3 MVP 目标

- 搭建完整的核心浏览链路：全国地图 -> 城市页 -> 景点页
- 产出一套统一风格的卡通化视觉内容
- 完成少量核心景点的轻量 3D 展示能力验证
- 为后续加入语音讲解、内部漫游、热点导览打下结构基础

## 3. MVP 范围

### 3.1 推荐首发范围

建议首版只覆盖 3 个城市，每个城市 2 到 4 个代表性景点。

推荐城市：

- 北京
- 上海
- 西安 或 杭州

推荐景点示例：

- 北京：故宫、天安门、天坛
- 上海：东方明珠、外滩、豫园
- 西安：兵马俑、大雁塔、钟楼
- 杭州：西湖、雷峰塔、灵隐寺

### 3.2 MVP 必做功能

- 全国卡通地图首页
- 点击城市进入城市景点总览页
- 城市页展示城市代表景点列表
- 点击景点进入景点详情页
- 景点详情页展示主视觉、简介和基础交互
- 至少 3 到 5 个景点支持轻量 3D 模型浏览

### 3.3 MVP 暂不纳入范围

- 全国所有城市全量覆盖
- 景点内部空间漫游
- 实时多人互动
- 社区评论和 UGC
- 完整路线规划
- 电商或票务购买
- 高精度写实 3D 扫描级建模

## 4. 用户核心流程

### 4.1 主流程

1. 用户打开 App，看到全国卡通地图
2. 用户点击某个已开通的城市
3. 进入城市页，浏览该城市的代表景点
4. 用户点击某个景点
5. 进入景点详情页，查看景点视觉、简介和 3D 展示
6. 用户返回继续浏览其他景点或切换到其他城市

### 4.2 首版用户价值

- 低门槛地“逛”中国代表城市和景点
- 快速获得一个景点的视觉印象和简短背景信息
- 感受到“赛博漫游”的新鲜感，而不只是图文百科

## 5. 页面与模块设计

### 5.1 全国地图首页

目标：
让用户愿意点开第一个城市。

页面要素：

- App 标题与简短副标题
- 全国卡通地图主视觉
- 已开放城市高亮显示
- 城市点击反馈
- 可选的“推荐城市”入口

交互说明：

- 未开放城市显示为不可点击或弱化状态
- 已开放城市点击后进入对应城市页
- 地图可支持轻量缩放或拖拽，但首版不必做复杂地图系统

### 5.2 城市景点总览页

目标：
承接地图点击后的浏览意图，让用户快速选中一个景点。

页面要素：

- 城市名称与城市视觉头图
- 城市一句话标签
- 景点卡片列表
- 景点封面图
- 景点名称
- 简短介绍

交互说明：

- 点击景点卡片进入景点详情页
- 支持返回地图或切换其他城市

### 5.3 景点详情页

目标：
展示景点的核心内容，并验证 3D 浏览的可行性。

页面要素：

- 景点名称
- 景点主视觉图
- 基础介绍文案
- 3D 模型展示区域
- 基础操作提示，例如旋转、缩放
- 可选的“相关景点推荐”

交互说明：

- 用户可旋转、缩放 3D 模型
- 如果景点尚未制作 3D，则展示卡通主视觉和说明信息
- 为后续的讲解音频、热点标签和内部漫游预留入口区域

## 6. 内容与资产规划

### 6.1 内容类型

MVP 内容主要包括：

- 全国卡通地图
- 城市头图
- 景点卡通主视觉
- 景点简要介绍文案
- 少量景点 3D 低模资源

### 6.2 建议的内容生产流程

1. 确定城市和景点名单
2. 为所有景点生成统一风格的卡通概念图
3. 从中挑选 3 到 5 个核心景点制作低模 3D
4. 为每个景点编写标准化简介文案
5. 在 App 内接入内容资源并验证展示效果

### 6.3 内容风格建议

- 视觉风格统一，避免不同城市像不同素材站拼接
- 优先“亲和、鲜明、可识别”，而不是写实
- 景点卡通化处理时保留地标轮廓特征
- 3D 模型控制面数和贴图大小，优先移动端流畅性

## 7. 功能需求清单

### 7.1 地图模块

- 展示全国主地图
- 标注已开放城市
- 支持城市点击跳转
- 支持当前已开放城市的视觉高亮

### 7.2 城市模块

- 展示城市信息
- 展示城市景点列表
- 支持景点入口跳转

### 7.3 景点模块

- 展示景点封面内容
- 展示景点介绍文案
- 展示 3D 模型或静态替代内容
- 支持模型基础交互

### 7.4 用户基础功能

- 浏览记录
- 收藏景点

说明：
如果开发资源有限，收藏与浏览记录可以放到 MVP 的末优先级，先确保主链路完成。

## 8. 非功能需求

### 8.1 性能

- 首屏加载快
- 地图页操作流畅
- 3D 模型在主流移动设备上可接受加载和交互

### 8.2 体验

- 页面视觉统一
- 交互路径简洁
- 用户在 3 次点击内能够进入任意一个已开放景点

### 8.3 可扩展性

- 内容结构能持续增加新城市和新景点
- 景点数据结构可以扩展音频、热点、内部场景等字段

## 9. 数据结构建议

建议至少规划三层内容对象：

### 9.1 城市对象

- 城市 ID
- 城市名称
- 地图坐标
- 城市头图
- 城市标签
- 景点列表
- 开放状态

### 9.2 景点对象

- 景点 ID
- 所属城市 ID
- 景点名称
- 封面图
- 简介
- 3D 资源地址
- 是否支持 3D
- 后续扩展字段：音频、内部场景、热点说明

### 9.3 用户行为对象

- 最近浏览景点
- 收藏景点列表

## 10. 验收标准

MVP 完成的最低标准：

- 用户可从全国地图进入至少 3 个城市
- 每个城市至少有 2 个景点可浏览
- 用户可进入景点详情页查看图文内容
- 至少 3 个景点支持基础 3D 浏览
- 页面风格统一，交互链路无阻断

## 11. 风险与控制

### 11.1 主要风险

- 内容制作量超预期
- 3D 资产制作周期过长
- 地图、美术、3D、前端同时推进导致失控
- 首版范围过大导致迟迟无法上线

### 11.2 控制策略

- 严格限制首发城市数量
- 所有景点先有卡通图，再挑少量做 3D
- 先做外观浏览，不做内部漫游
- 优先跑通主链路，再补充收藏、音频等增强功能

## 12. 下一步建议

建议按以下顺序推进：

1. 确定首批城市和景点名单
2. 输出低保真原型图
3. 确定卡通视觉风格
4. 建立景点内容生产规范
5. 选择技术方案并进入开发
